数据库系统概念第十一章——索引与HASH

1. 索引的概念

​ 索引类似于书本的索引。比如，为了得到给定ID的student记录，数据库先查找索引，找到相应记录所在的磁盘块，然后取出该磁盘块，得到所需记录。

​ 一般来说，一张表上会有多个索引。比如说，可能根据“作者”、“主题”、“书名”来查找一本书。

​ “作者”、“主题”、“书名”这些用来查找数据的信息又被称为“搜索码”（search key）。

1.1 分类

​ 有两种基本的索引类型：

  1. 顺序索引
  2. 散列索引

1.2 评价因素

​ 没有哪一个索引是最好的，好坏取决于应用场景，评价应当基于以下因素：

  1. 访问类型
  2. 访问时间
  3. 插入/删除时间
  4. 空间开销

2. 顺序索引

2.1 概念

​ 聚集索引（又叫做主索引）：通常建立在表的主键上，它要保证建立索引的属性必须有序，但并非一定建立在主键上。

​ 辅助索引（Secondary Index）：又被称为”非聚集索引“。

​ 索引顺序文件（index-sequential file）：文件按照搜索码顺序排列，在搜索码上有聚集索引。

2.2 稀疏索引和稠密索引

2.2.1 稠密索引

​ 稠密索引是指：文件中的每一个搜素码值都有一个索引项与之对应。

​ 如下图，假设搜索码是工号，在工号上建立了一个稠密索引。

​

​ 如上图，我们设搜索码是系名，为每个系名建立稠密索引。

​ 稠密索引会导致索引文件特别大，记录庞大的情况下，很少采用这种方法。

2.2.2 稀疏索引

​ 只为搜索码的部分值建立索引，而且必须建立在聚集索引上（为了保证有序！）

​

​

​ 比如搜索码是工号，我们只为3个值建立索引。在搜索时，相当于将整个表划分为3块$(10101~22222)$、$(32343,76543)$、$(76766,98345)$，先查找索引，定位工号在哪块，然后在对应块中做顺序搜索即可。

2.2.3 辅助索引

​ 辅助索引必须是稠密的，因为辅助索引上的搜索码并不按照顺序排列。

​ 例如，我们在薪水属性上增加一个辅助索引。

2.3 多级索引

​ 如果元组数目过多，且在上面建立稠密索引进行搜索，索引所占用的空间可能会非常大，以至于不能放到内存中。因此我们可以建立多级索引去缩小索引的大小。（外层建稀疏索引，内层建稠密索引）

​

2.4 插入和删除

​ 如果原表中的记录被插入/删除，那么索引也要跟着进行维护。

3. B+树索引文件

3.1 B+树的定义

​ B+树应当满足如下4个性质：

  1. 从根节点到每个叶子结点路径长度相同
  2. 非根非叶子结点有$\lceil n/2 \rceil$到$n$个孩子
  3. 叶子结点有$\lceil (n-1)/2 \rceil$到$n-1$个值
  4. 如果整个树只有一个根节点，那么它可以有任意个孩子；如果树的根不是叶子，那么它必须有至少2个孩子

​ 注：B+树的高度为$\lceil log_{\lceil n/2 \rceil}(K)\rceil$

3.2 B+树的结点结构

​ $K_i$是搜索码，$P_i$是指针（指向孩子或者记录）。

​ $K_i$必须有序。

3.2.1 B+树的叶子结点结构

​ 叶子结点的指针指向一条文件记录，结点的最后一个指针指向下一个B+树的叶子结点，如下图所示：

3.2.2 B+树非叶子结点结构

​ 对于非叶子结点，结点左边的指针指向比它小的记录，右边指针指向大于等于它（注意等于）的记录。

​ 例如上图，El Said左侧指针指向的Brandt、Califieri等结点均字典序小于El Said，右侧指向的结点包含了El Said、Gold等大于等于它字典序的结点。

​ 在$n=6$时，每个叶子结点必须有$3…5$个结点。

3.3 B+树的查找

效率分析

​ 结点大小一般等于磁盘块大小（4KB），n一般设为100。在100万条记录的情况下，树的高度也仅为4。

重复搜索码的处理

​ 如果搜索码不唯一，可以给搜索码添加主键属性，使它变得唯一。

​ 如果要查找k=v的所有记录，可以将这个问题转换为一个简单的区间查询问题。

3.4 B+树的插入

​ 基本思想：找到插入叶结点位置，插入后，若违反了B+树的性质（结点已满），则向上分裂以保持B+树的性质。

​ 如果结点已满，那么提取中间结点，将它提到父节点，如果父节点已满，那么继续将父节点的中间结点向上提，以此类推。

3.5 B+树的删除

​ 首先查询要删除的结点位置（叶结点），然后移除它，并将它右侧的所有项都左移一个位置。如果出现结点过少的情况，则要向兄弟结点去“借“项或合并两个结点，使得树的结构符合B+树的性质。

**3.6 复杂度分析

​ 可以证明，B+树的插入和删除操作的复杂度和树的高度$\lceil log_{\lceil n/2 \rceil}(K)\rceil$成正比。

​ 在实际应用中，B+树的插入和删除复杂度都是$O(1)$的，因为插入删除造成树结构调整的情况并不多。

3.7 和B-树的区别

​ B+树的实际数据都存储在叶子结点（非叶子结点相当于索引，因此每个值在树中可能出现多次），而B-树的实际数据可以在非叶子结点出现（每个值在树中仅出现一次）。

  • B-树的优点

    B-树的高度小于B+树。

    搜索时如果在中间结点找到目标结点，可以直接返回。

4. HASH

4.1 概念

​ hash索引：桶存储指向记录（record）的指针。

​ hash文件组织：桶直接存储记录。

​ 静态hash：指桶的数量不变。

​ 动态hash：桶的数量会随着记录的增加/减少而动态变化。

​ 开地址hashing：线性/平方探测法。

​ 闭地址hashing：经典的拉链法。

4.2 静态hash

​ 静态hash会在存放之前就固定桶的个数，因此这会导致两个问题：

  1. 如果桶分配得太多，那么会造成空间浪费。
  2. 如果桶分配得太少，随着记录不断增加，冲突也会越来越多。

​ 这个问题可以通过重构（reorganize解决），当桶不足时，增加桶的数量，并重新设置hash函数。

4.3 动态hash

​ 当记录个数大于桶个数的1.5倍时，创建2倍桶的hash表，然后将原hash表数据映射到新表内。

5. 位图索引

​ 将表中索引进行重新编号$1…n$

​ 对每一个属性的每一个值都建一个位图（bitmap），其长度为元组个数。

如上图所示，对gender的f属性和m属性分别创建位图索引

  • 位图索引优势

    1.占用空间很小。

    2.在做AND/OR类型的查询时，速度极快（因为转化成了位运算）。比如要求出gender=m且income_level=L1的人，可以直接用m的位图对L1的位图做交运算即可。

  • 位图索引劣势

    1.只适用于属性种数少的情况。
